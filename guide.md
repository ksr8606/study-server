# 앱 개발만 하던 너에게 — 서버 개발 입문 가이드

> 같은 팀 서버 개발자가 쓴 글이야. 앱(클라이언트)만 만지다 서버로 넘어오면
> "어? 이건 왜 이렇게 하지?" 싶은 게 한두 개가 아니야.
> 문법보다 **사고방식의 차이**부터 잡아주는 게 목적이다. 천천히 읽어.

---

## 0. 가장 큰 차이 한 줄 요약

**앱은 "한 사람"을 위해 돌고, 서버는 "동시에 수천 명"을 위해 돈다.**

이 한 줄이 모든 차이의 뿌리야.
앱에서는 화면 하나, 유저 한 명, 그 사람의 상태만 신경 쓰면 됐지.
서버는 같은 코드가 동시에 수천 번 실행돼. 이 감각이 안 잡히면 계속 헤맨다.

---

## 1. 상태(State)를 바라보는 관점이 완전히 다르다

### 앱에서 너는
- 화면에 보이는 상태를 메모리에 들고 있었음 (ViewModel, 전역 싱글톤 등)
- 앱이 켜져 있는 동안 그 상태는 계속 살아있음
- "유저 = 나 한 명"이라 충돌 걱정이 없었음

### 서버에서는
- **기본적으로 stateless(무상태)를 지향한다.** 요청 하나 처리하고 나면 그 메모리는 날아간다고 생각해.
- 상태는 메모리가 아니라 **DB / 캐시(Redis 등)** 에 둔다.
- 서버 인스턴스가 여러 대(스케일 아웃)일 수 있어서, A서버에 저장한 메모리 값은 B서버가 모름.

> ⚠️ 신입이 제일 많이 하는 실수: "전역 변수에 유저 데이터 캐싱해두면 빠르겠지?"
> → 서버 2대로 늘어나는 순간 데이터 불일치로 터진다. 상태는 외부 저장소에 둬라.

---

## 2. 동시성(Concurrency) — 이게 진짜 새로운 세계다

같은 함수가 **동시에** 실행된다. 이걸 항상 머릿속에 깔고 있어야 해.

```
유저 A: 잔액 1000원 읽음 → -500 계산 → 500원 저장
유저 B: 잔액 1000원 읽음 → -500 계산 → 500원 저장   (거의 동시에)
결과: 1000원이 두 번 빠졌는데 잔액은 500원   ← 돈이 사라짐
```

이걸 **레이스 컨디션(race condition)** 이라고 해. 앱에선 거의 만날 일 없던 버그야.

해결 도구들:
- **DB 트랜잭션 + 락(lock)**: 가장 기본. "이 줄 건드리는 동안 너 기다려."
- **낙관적 락 / 비관적 락**: 충돌이 드문지 잦은지에 따라 선택.
- **원자적 연산**: `UPDATE balance = balance - 500 WHERE ...` 처럼 읽고-쓰고를 한 방에.

> 핵심: "내 코드가 한 줄씩 순서대로 실행될 거다"라는 앱 시절의 가정을 버려라.

---

## 3. 데이터베이스가 너의 새 무대다

앱에서 로컬 DB(SQLite, Room, CoreData) 좀 써봤겠지만, 서버에선 DB가 **주인공**이야.

알아둬야 할 것:
- **인덱스(Index)**: 없으면 100만 건을 매번 처음부터 다 뒤진다. 느린 쿼리의 90%는 인덱스 문제.
- **N+1 문제**: 목록 100개 가져오고, 각각에 대해 쿼리 100번 더 날리는 것. ORM 쓰면 자기도 모르게 이걸 한다. 항상 의심해.
- **트랜잭션**: 여러 작업을 "전부 성공 아니면 전부 취소"로 묶는 것. 송금에서 출금만 되고 입금이 안 되면 안 되잖아.
- **정규화 vs 비정규화**: 중복을 줄일지(정규화), 속도를 위해 일부러 중복할지(비정규화)의 트레이드오프.

> 실전 팁: 쿼리 짤 때 항상 "데이터 1000만 건이어도 이게 빠를까?"를 자문해.
> 개발 DB엔 10건 있어서 다 빨라 보인다. 운영에선 안 그래.

---

## 4. API 설계 — 네가 이제 "제공자"다

앱 할 땐 서버 API를 **갖다 썼지만**, 이제 네가 **만들어 주는** 쪽이야.
앱 개발자(과거의 너)가 쓰기 편하게 만들어야 해.

좋은 API의 조건:
- **일관성**: 응답 형식, 에러 형식, 네이밍이 엔드포인트마다 들쭉날쭉하면 클라가 미친다.
- **명확한 HTTP 상태 코드**: 200(성공), 400(요청 잘못), 401(인증 안 됨), 403(권한 없음), 404(없음), 500(서버 잘못). 다 200으로 주고 body에 에러 넣는 짓 하지 마.
- **버저닝**: `/v1/users` 처럼. 한 번 배포된 API는 쓰는 앱들이 있어서 함부로 못 바꾼다.
- **하위 호환성**: 필드 삭제/이름 변경은 구버전 앱을 깨뜨린다. 이게 앱과의 결정적 차이 — **너는 클라를 강제로 업데이트 못 시킨다.**

> 앱은 새 버전 배포하면 끝이지만, 서버 API는 옛날 앱들이 계속 때린다.
> "옛날 클라가 아직 이 API 쓰고 있나?"를 항상 생각해야 해.

---

## 5. 인증(Authentication) vs 인가(Authorization)

- **인증**: "너 누구야?" → 로그인, 토큰(JWT 등)
- **인가**: "너 이거 할 권한 있어?" → 남의 글을 네가 삭제하려 하면 막아야지

> 신입 단골 사고: 인증만 하고 인가를 안 함.
> `DELETE /posts/123` 요청에서 "로그인했네 OK" 하고 끝내면,
> 123번 글이 **남의 글**이어도 지워진다. 요청한 유저가 그 자원의 주인인지 반드시 확인해.

보안 기본기 (앱보다 훨씬 빡세게 봐야 함):
- 클라가 보낸 값은 **전부 의심**해라. (입력 검증, SQL 인젝션, XSS)
- 비밀번호는 평문 저장 절대 금지. 해시(bcrypt 등)로.
- 민감 정보를 로그에 찍지 마라.

---

## 6. 에러를 대하는 자세

앱에서 에러나면 그 사람 화면만 깨졌어. 서버에서 에러나면 **수천 명이 동시에** 영향받아.

- **모든 외부 호출은 실패할 수 있다**: DB 연결, 다른 서버 API 호출, 네트워크. 항상 실패를 가정하고 짜.
- **에러를 삼키지 마라**: `try { } catch { /* 무시 */ }` 는 운영에서 원인 못 찾는 지옥의 시작.
- **타임아웃을 걸어라**: 응답 없는 외부 호출을 무한정 기다리면 그 스레드/커넥션이 묶여서 서버 전체가 마비된다.

---

## 7. 관측 가능성(Observability) — 안 보이면 못 고친다

앱은 디버거 붙여서 한 스텝씩 보면 됐지. 서버는 운영 중에 디버거 못 붙여. 대신:

- **로깅**: 무슨 일이 있었는지 기록. 단, 너무 많이 찍으면 정작 중요한 게 묻힌다. 레벨(DEBUG/INFO/WARN/ERROR) 구분해.
- **메트릭**: 요청 수, 응답 시간, 에러율 같은 숫자 추이. (Prometheus, Grafana 등)
- **추적(Tracing)**: 요청 하나가 여러 서버를 거칠 때 흐름 따라가기.

> "운영에서 문제 터졌는데 로그를 안 남겨놔서 원인을 모르겠다"
> — 이게 서버 개발자 최악의 악몽이다. 로그는 미래의 너에게 보내는 편지야.

---

## 8. 배포와 운영 — 끝이 아니라 시작

앱은 스토어 심사 통과하면 손 떠났지. 서버는 **네가 24시간 책임진다.**

- **무중단 배포**: 서비스 안 끊고 새 버전 올리기 (롤링, 블루-그린 등)
- **롤백**: 새 버전이 터지면 즉시 이전 버전으로 되돌리기. 항상 되돌릴 수 있게 준비해둬.
- **마이그레이션**: DB 스키마 변경은 신중하게. 한 번 잘못 날린 마이그레이션은 데이터를 영구히 날린다.
- **환경 분리**: 로컬 / 개발 / 스테이징 / 운영. 운영 DB에 직접 쿼리 날리는 건 극도로 조심.

---

## 9. 성능을 보는 눈

| 앱에서 신경 쓰던 것 | 서버에서 신경 쓰는 것 |
|---|---|
| 프레임 드랍, UI 렉 | 처리량(throughput), 응답 시간(latency) |
| 앱 용량, 메모리 | 동시 접속, DB 커넥션 풀 |
| 배터리 | 서버 비용, CPU/메모리 |

도구:
- **캐싱**: 자주 읽고 잘 안 바뀌는 데이터는 Redis 등에 캐싱. 단, "캐시 무효화"가 어렵다는 걸 명심.
- **비동기 처리 / 큐**: 무거운 작업(이메일 발송, 이미지 처리)은 요청 응답에서 분리해서 큐로 넘겨.
- **부하 테스트**: 배포 전에 "초당 1000요청 버티나?" 미리 때려보기.

---

## 10. 마지막 — 마인드셋 정리

1. **항상 "동시에 수천 명"을 상상해라.** 내 코드가 한 번만 도는 게 아니다.
2. **모든 것은 실패한다고 가정해라.** DB도, 네트워크도, 외부 API도.
3. **상태는 외부에 둬라.** 메모리를 믿지 마.
4. **클라이언트(앱)를 절대 신뢰하지 마라.** 검증은 서버에서.
5. **미래의 디버깅을 위해 로그를 남겨라.**
6. **되돌릴 수 있게 만들어라.** 배포도, 데이터 변경도.

### 추천 학습 순서
1. HTTP / REST 기본기 (상태 코드, 메서드, 헤더)
2. SQL과 DB 인덱스 — 이거 제대로 하면 중급은 먹고 들어간다
3. 트랜잭션과 동시성
4. 인증/인가 + 보안 기본
5. 캐싱, 비동기, 메시지 큐
6. 배포/모니터링/장애 대응

모르는 거 있으면 혼자 끙끙대지 말고 바로 물어봐.
앱에서 쌓은 경험(사용자 입장에서 생각하기, 클라가 뭘 원하는지 아는 것)은
서버에서도 엄청난 무기다. API 설계할 때 그 감각이 빛을 발할 거야.

환영한다. 같이 잘 해보자. 🚀
